# 演员头像数据格式说明（Filetree.json + Content）

本应用通过读取**演员数据根目录**下的 `Filetree.json` 与 `Content` 目录来匹配并展示演员头像。数据来源可与 [gfriends/gfriends](https://github.com/gfriends/gfriends) 项目一致；若你使用自建或第三方数据，需符合下述格式以便应用正确识别。

---

## 一、目录结构要求

演员数据根目录下**必须**包含：

- **Filetree.json**：描述「演员显示名 → 图片文件」的映射关系。
- **Content**：存放图片的文件夹；其下按「分组」再分子目录，每个子目录内为实际图片文件。

```
演员数据根目录/
├── Filetree.json
└── Content/
    ├── 分组名A/
    │   ├── 实际图片1.jpg
    │   └── 实际图片2.png
    └── 分组名B/
        └── 实际图片3.jpg
```

应用会校验：根目录下存在 `Filetree.json` 且存在 `Content` 目录，才会允许将该路径设为「演员数据路径」。

---

## 二、Filetree.json 格式

### 2.1 顶层结构

- 根对象中**必须**包含键 `Content` 或 `content`（大小写按此为准，本应用两种都会识别）。
- 其值为一个**对象**，该对象的键为**分组名**，值同样为对象（见下）。

示例：

```json
{
  "Content": {
    "泰坦尼克号": { ... },
    "肖申克的救赎": { ... }
  }
}
```

### 2.2 分组内的映射（演员名 → 图片文件）

在 `Content` 下的每个分组对象中：

- **键（key）**：表示**演员显示名 + 扩展名**，例如 `莱昂纳多·迪卡普里奥.jpg`。  
  - 应用会用「去掉扩展名」后的部分作为**演员名**参与匹配（如 `莱昂纳多·迪卡普里奥`）。  
  - 扩展名仅用于从键中解析演员名，不影响实际文件格式。
- **值（value）**：**字符串**，表示该分组目录下的**实际图片文件名**。  
  - 若带查询串（如 `?t=1607433809`），应用会**只取 `?` 之前的部分**作为文件名。  
  - 例如：`"莱昂纳多·迪卡普里奥.jpg?t=1607433809"` → 实际读取文件名为 `莱昂纳多·迪卡普里奥.jpg`。

示例（与 gfriends 风格一致）：

```json
{
  "Content": {
    "泰坦尼克号": {  //文件夹名称无实际意义，可按自己的喜好分类，如首字母排序等，只要与实际文件路径保持一致既可
      "莱昂纳多·迪卡普里奥.jpg": "莱昂纳多·迪卡普里奥.jpg?t=1607433809",
      "凯特·温斯莱特.jpg": "凯特·温斯莱特.jpg?t=1607433809",
      "Kate Winslet.jpg": "凯特·温斯莱特.jpg?t=1607433809"
    }
  }
}
```

- 键 `莱昂纳多·迪卡普里奥.jpg` → 演员名 `莱昂纳多·迪卡普里奥`，对应图片文件 `Content/泰坦尼克号/莱昂纳多·迪卡普里奥.jpg`。
- 键 `凯特·温斯莱特.jpg` 与 `Kate Winslet.jpg` 指向同一张图，应用会合并为同一演员的多个「候选头像」。

### 2.3 规则小结

| 项目       | 说明 |
|------------|------|
| 键（key）  | 任意「演员显示名.扩展名」，扩展名通常为 `.jpg` / `.png` 等；用于匹配的演员名 = 去掉扩展名后的字符串。 |
| 值（value）| 该分组下**实际存在的图片文件名**。可带可选后缀 `?xxx`（如 gfriends 的 `?t=1607433809`），应用会忽略 `?` 及之后部分，仅用前面作为文件名。 |
| 分组名     | 与 `Content` 下子目录名**一一对应**，用于拼出路径：`Content/分组名/实际文件名`。 |

---

## 三、Content 目录与图片文件

### 3.1 目录对应关系

- `Content` 必须位于**演员数据根目录**下。
- `Content` 下的**子目录名**必须与 `Filetree.json` 里 `Content` 下的**分组名**一致（如 `泰坦尼克号`）。
- 每个子目录内放置该分组在 Filetree 中**值（value）**所列出的图片文件（取 `?` 前部分作为文件名）。

因此，实际图片路径为：

```
<演员数据根目录>/Content/<分组名>/<实际文件名>
```

例如根目录为 `F:\movie\actors` 时：

- `F:\movie\actors\Content\泰坦尼克号\莱昂纳多·迪卡普里奥.jpg`

### 3.2 图片格式

- 支持常见图片格式：**.jpg / .jpeg / .png / .webp / .gif**。
- 文件必须真实存在且可读；若文件名或路径与 Filetree 不一致，该条在应用中会无法显示或报错。

---

## 四、应用内的匹配与使用

### 4.1 演员名匹配

- 应用使用 NFO/数据库中的**演员名**与 Filetree 中**键去掉扩展名**后的字符串进行匹配。
- 支持**简繁体互通**：若库中为简体、Filetree 为繁体（或反之），应用会做转换后再匹配。
- 若在应用中为演员设置了**显示名**或**曾用名**，执行「扫描演员信息」时，这些名称也会参与匹配，并合并到该演员的候选头像中。

### 4.2 多头像与默认图

- 同一演员可对应多条 Filetree 映射（不同键指向同一或不同图片），应用会汇总为**候选头像列表**。
- 用户可在应用内**选择演员头像**中指定默认展示哪一张；该选择会保存在应用本地，与 Filetree/Content 文件无关。

### 4.3 扫描流程

1. 在**设置**中选择演员数据根目录（须含 `Filetree.json` 与 `Content`）。
2. 点击「**扫描演员信息**」。
3. 应用读取 `Filetree.json`，按上述规则解析并建立「演员名 → 图片路径」映射，并与本地库中演员/显示名/曾用名匹配。
4. 扫描结果写入应用本地映射（如 `actor-avatar-map.json`），后续列表与详情页按此显示头像。

---

## 五、自定义与修改建议

若你**自行维护** Filetree.json 或 Content 目录：

1. **保持结构**：根目录下始终有 `Filetree.json` 和 `Content`，且 `Content` 下子目录名与 Filetree 中分组名一致。
2. **键的命名**：键建议使用「演员名 + 统一扩展名」（如 `.jpg`），演员名尽量与 NFO/刮削器中的写法一致或互为简繁体，便于自动匹配。
3. **值的命名**：值为实际文件名，需与 `Content/<分组名>/` 下真实文件名一致（忽略 `?` 及之后部分）。
4. **增删改后**：在应用中重新执行「扫描演员信息」即可刷新头像映射。
5. **编码**：Filetree.json 建议使用 **UTF-8** 保存，避免日文、繁体等出现乱码。

**简单来说**，JSON文件的**左边**代表数据库中的**演员名称**，需与数据库名称一致，**右边**代表与之对应的**图片名称**，需与实际图片文件名称一致；左右边的**名称**无必然关系，仅代表匹配映射关系；JSON数据的层级关系，代表了图片的查找路径。为方便大家自建库，可以在自行搭建好图片库后，使用附带的`Filetree文件生成工具.exe`进行快速生成`Filetree.json`文件


以上格式与 gfriends 示例（如 `F:\movie\actors`）兼容；按此说明组织或修改数据后，即可在本应用中正常使用演员头像和演员信息编辑功能。
